01 — Problem

The Challenge

Bluesky's decentralised architecture — built on the AT Protocol — distributes identity hosting, moderation, and feed ranking across independent actors. This modularity serves the platform's decentralisation goals, but the interface deliberately mirrors Twitter/X to ease onboarding — concealing real architectural complexity underneath.

The gap between intent and behaviour

Bluesky positions itself for casual users, creators, and power users alike. But adoption data tells a different story: 98.9% of users rely on the default identity provider, and only 1.1% adopt decentralised identity hosting. Most people want convenience, not autonomy — yet the interface is built as if everyone wants both.

Evaluation Scope

This evaluation examined Bluesky's homepage and core interactions across three lenses using empirical tools rather than opinion:

  • Usability — Nielsen's Heuristic Evaluation across three personas
  • Accessibility — WCAG 2.2 audit via AccessibilityChecker.org and WAVE
  • Sustainability — Website Carbon Calculator, Lighthouse, Wayback Machine, hosting-stack scans
02 — User Lens

Personas & Behaviour

Three personas were evaluated against Nielsen's usability heuristics to surface how Bluesky's decentralised affordances are actually experienced, versus how they're designed to be experienced.

Persona 01 —

Nikita, Creator

Pain point: No analytics, reach indicators, or insight into how posts travel across feeds.

Heuristics violated: Visibility of system status; match between system and real-world expectations.

Proposed Solution

Simple, anonymous reach indicators — e.g. "your post appeared in 12 feeds today" — based on aggregated feed-generator counts. Gives creators feedback without centralising data or copying platform-style analytics.

Persona 02 —

Ben, Regular Consumer

Pain point: Feed discovery is cumbersome; many feeds feel unreliable or low quality.

Heuristics violated: Consistency and standards; recognition rather than recall.

Proposed Solution

Feed metadata — update frequency, ranking logic, operator identity — similar to how Matrix clients reveal which homeserver routes traffic. Helps users choose trustworthy sources without added cognitive load.

Persona 03 —

Alex, Power User

Pain point: Information overload from multiple feeds and moderation tools; inconsistent label vocabularies.

Heuristics violated: Consistency and standards; user control and freedom; error prevention.

Proposed Solution

A unified moderation and feed dashboard that visually groups active feeds, labelers, mutelists, and blocklists, and flags overlaps or conflicts — centralising visibility, not moderation, so power users can manage complexity without juggling disjoint interfaces.

03 — Usability Evaluation

Heuristic Findings

Using Nielsen's Heuristic Evaluation framework across all three personas, the same root cause kept recurring: decentralisation requires conceptual knowledge most users don't have, and the interface offers no scaffolding for it.

  • Creators get no feedback loop on reach or visibility
  • Feed discovery relies on third-party tools, not the platform itself
  • Moderation and labelling tools use inconsistent vocabulary
  • Recognition-over-recall failures increase cognitive load for all three personas
Design Recommendations
  • Anonymous creator reach indicators
  • Transparent feed metadata (update frequency, ranking logic, operator)
  • Unified moderation & feed visibility dashboard

Security & Data Handling

Bluesky uses HTTPS and secure login, but decentralisation complicates privacy expectations. Relay and AppView services act as centralisation bottlenecks — single points of failure in an otherwise distributed system.

In November 2024, roughly 1 million public posts were scraped and reused for AI training without user consent — exposing a governance gap in public-by-default APIs that decentralisation alone doesn't solve.

04 — Accessibility Audit

WCAG 2.2 Compliance

23%

Accessibility score via AccessibilityChecker.org — well below WCAG 2.2 Level AA compliance.

233+

Critical issues identified in the audit. The volume and pattern indicate substantial failures.

0

Published accessibility statements — no documentation of compliance level or known barriers exists.

The audit revealed accessibility problems that are structural, not cosmetic:

  • Empty aria-label attributes on key controls — fails WCAG 4.1.2
  • Like/repost counters in grey-on-grey, below 4.5:1 contrast minimum — fails WCAG 1.4.3
  • Editorial images in news previews missing alt text — fails WCAG 1.1.1
  • No semantic landmarks (<main>, <nav>, <aside>) — forces linear navigation for assistive tech
  • Missing lang attribute on <html> — fails WCAG 3.1.1

Mapped back to the personas, these failures translate into broken journeys, not abstract violations:

  • Ben hits friction from unlabeled controls and low-contrast text
  • Alex can compensate via device-level settings, but gets no in-app guidance
  • Nikita can't tell how accessible her own content is to the audiences she's trying to reach

Accessibility on Bluesky currently depends on user expertise rather than platform responsibility — which undercuts the inclusivity a decentralised, open platform claims to offer.

05 — Sustainability & Performance

Environmental Evaluation

Grade F

Website Carbon Calculator rating — Bluesky's homepage produces more CO₂ per visit than roughly 84% of tested sites.

2022 →

Earliest Wayback Machine capture — the homepage has a stable archival presence, though JS-heavy single-page apps are often only partially preserved.

No Evidence

Result of a hosting-stack transparency scan — no CDN or server metadata is publicly detectable, so energy/carbon claims can't be verified.

Lighthouse runs (mobile and desktop) consistently surfaced the same issues:

  • Unused JavaScript and render-blocking resources
  • Large Cumulative Layout Shift (CLS)
  • Inefficient caching policies
  • Unoptimised images and unused CSS

These compound into higher client-side CPU usage and data transfer — greater energy draw, especially on low-power mobile hardware.

Long-term Viability

A broken-link check found no significant failures — a positive sign for structural stability. But other signals point the other way:

  • Significant CLS suggests structural volatility undermining both accessibility and archival fidelity
  • Heavy JS reliance with large unused payloads increases technical debt over time
  • No publicly documented strategy for longevity, maintainability, or sustainable hosting
06 — Technical Validation

Structural Issues

Critical Issue

W3C validator analysis found the <html> tag lacks a language declaration, preventing screen readers from applying correct pronunciation rules — a real barrier for visually impaired users.

Fix

<html lang="en">

Five further markup errors were identified:

  • Missing content in <meta name="theme-color">
  • Empty href="" in the ActivityPub link
  • Missing rel attribute on a link element
  • Unclosed <noscript> tags

The real takeaway: Bluesky's substantive UX problems are conceptual, not syntactic — they live in information architecture, not HTML compliance.

07 — Recommendations

Design Actions

Rec 01 —

Onboarding & Transparency

Improve onboarding around decentralised concepts and increase transparency of feeds, ranking logic, and moderation systems.

Rec 02 —

Accessibility & Semantics

Strengthen semantic HTML, address WCAG compliance gaps, and publish accessibility documentation for accountability.

Rec 03 —

Performance & Governance

Optimise front-end resources, reduce unused scripts, and increase infrastructure transparency and governance clarity.

08 — Reflection

Takeaways

Bluesky's evaluation raises a question bigger than one platform: can decentralised systems stay usable, accessible, and sustainable as they scale — or does technical openness inevitably trade off against ease of use?

The adoption data is the clearest signal: 98.9% of users default to the path of least resistance. Autonomy that requires expertise to access isn't really autonomy for most people. For decentralisation to fulfil its democratic promise, technical openness needs to be matched with clearer onboarding, transparency layers, and accessible-by-default scaffolding.
